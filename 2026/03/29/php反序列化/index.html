PHP反序列化

一、PHP 反序列化基础原理

1.1 序列化与反序列化的概念

在 PHP 中，序列化 (Serialization)是将对象的状态信息转化为可存储或传输的形式（通常是字符串）的过程。而反序列化 (Deserialization) 则是将序列化后的字符串重新转换回对象的过程。这两个过程相辅相成，为数据的存储和传输提供了便利。

PHP 提供了两个主要函数来处理序列化和反序列化：

  • serialize()：将对象转换为字符串

  • unserialize()：将字符串转换回对象

序列化的主要目的是：

  1. 将对象状态保存在文件或数据库中

  2. 将对象进行远程传输（如前后端交互发送数据）

1.5 魔术方法在反序列化中的作用

PHP 提供了一系列魔术方法，这些方法会在特定操作时自动调用。在反序列化过程中，以下魔术方法特别重要：

魔术方法 触发时机
__construct() 对象创建时调用
__destruct() 对象销毁时调用
__sleep() 序列化前调用
__wakeup() 反序列化后调用
__toString() 对象被当作字符串使用时调用
__invoke() 对象被当作函数调用时调用

这些魔术方法为攻击者提供了潜在的攻击点，特别是当这些方法中包含危险操作（如文件操作、数据库查询或代码执行）时。

二、反序列化漏洞原理与基础利用

2.1 反序列化漏洞的本质

反序列化漏洞的本质是程序在对用户提供的序列化数据进行反序列化时，没有进行充分的验证和过滤，导致攻击者可以控制反序列化过程，进而执行恶意代码或进行其他恶意操作。

漏洞产生的条件：

  1. 应用程序使用unserialize()函数处理用户输入

  2. 输入数据未经过充分验证

  3. 存在可利用的魔术方法或对象方法调用链

简单来说，当攻击者能够控制unserialize()的输入时，就有可能构造恶意的序列化字符串，在反序列化过程中触发一系列方法调用，最终导致安全问题。

三、POP 链构造与利用

3.1 POP 链的基本概念

POP 链（Property-Oriented Programming Chain，面向属性编程链）是反序列化漏洞利用的核心技术。其核心思想是通过控制对象的属性值，引导程序执行流经过多个类的方法调用，最终触发危险操作（如代码执行、文件读写等）。

POP 链的构建需要以下几个关键步骤：

  1. 寻找可利用的魔术方法作为起点

  2. 构建方法调用链，连接起点和终点

  3. 找到危险函数或操作作为终点

3.4 基于 PHAR 文件的 POP 链利用

PHAR（PHP ARchive）是 PHP 中类似于 JAR 的一种打包文件格式，它可以存储用户自定义的元数据，并且这些元数据是以序列化形式存储的。这一特性使得 PHAR 文件成为反序列化攻击的重要载体。

PHAR 文件的结构：

  • Stub：文件头，必须以<?php xxx __HALT_COMPILER(); ?>结尾

  • Manifest：包含文件权限、属性以及序列化形式存储的用户自定义元数据

  • Contents：实际的文件内容

  • Signature（可选）：验证文件完整性和真实性

当使用文件操作函数处理 PHAR 文件时，会自动触发元数据的反序列化过程。支持 PHAR 协议的函数包括：

  • fileatime, filectime, file_exists, file_get_contents

  • file_put_contents, file, filegroup, fopen

  • fileinode, filemtime, fileowner, fileperms

  • is_dir, is_executable, is_file, is_link

  • is_readable, is_writable, is_writeable, parse_ini_file

  • copy, unlink, stat, readfile 等

4.7 PHP 版本差异与兼容性问题

不同 PHP 版本对反序列化的处理存在差异，这可能影响 POP 链的有效性：

1. PHP 5.6.x 与 7.x 的差异

  • 在 PHP 5.6.24 之前的版本中，存在通过在对象长度前添加+号绕过正则过滤的漏洞

  • PHP 7.x 引入了更严格的类型检查，可能影响某些类型的 POP 链构造

2. PHP 8.x 的变化

  • PHP 8.0 引入了更严格的错误处理，可能改变某些魔术方法的触发条件

  • PHP 8.4 将默认的 bcrypt 加密级别从 10 提高到 12，增强了密码哈希的强度

  • PHP 9.0 计划引入更严格的反序列化异常处理，当unserialize()失败时抛出异常，而不仅仅是发出警告
